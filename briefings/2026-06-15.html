The only scalable delete in Postgres is DROP TABLE
172 pts · 63 comments · HN discussion · → Read more
Large DELETE operations in Postgres don't scale: they leave behind dead tuples that bloat tables, add replication overhead, and never reclaim disk space until a vacuum runs. The fix is structural — use date-based partitioning so bulk cleanup becomes an instant DROP TABLE or TRUNCATE instead of a row-by-row DELETE, since both operations remove files at the OS level independent of data size. It's a useful operational checklist for anyone running retention jobs against large Postgres tables.
